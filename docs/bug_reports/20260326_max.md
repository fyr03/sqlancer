# MySQL Bug Report
## Title
Incorrect result (NULL included) in index lookup/MAX() on FLOAT ZEROFILL column when filtering by 0.0

## Category
Server: Optimizer / InnoDB

## Version
mysql  Ver 9.6.0 for Linux on x86_64 (conda-forge)

## Description
When a FLOAT ZEROFILL column has a secondary index and contains a mix of zeros (including those coerced from strings) and NULL values, a query filtering for c0 = 0.0 incorrectly includes a NULL value in the result set when using a Covering Index Lookup.

This leads to incorrect results for aggregate functions like MAX(c0), which returns NULL instead of 0.0. The issue disappears if the index is ignored (IGNORE INDEX), suggesting a bug in the index range scan or boundary detection logic.

## Reproduction Steps
```SQL
-- 1. Setup table with FLOAT ZEROFILL and a secondary index
DROP TABLE IF EXISTS mysql_bug_mre;
CREATE TABLE mysql_bug_mre (
    c0 FLOAT ZEROFILL NULL
) CHARACTER SET utf8mb4 COLLATE utf8mb4_bin;
CREATE INDEX i_c0 ON mysql_bug_mre (c0);

-- 2. Insert mix of zeros, coerced strings, and NULLs
INSERT IGNORE INTO mysql_bug_mre (c0) VALUES (0.0), ("x"), (NULL);

-- 3. Expand data to trigger optimizer statistics change
INSERT IGNORE INTO mysql_bug_mre (c0) VALUES 
(0.0), (0.0), (0.0), (""), ("x"), ("-0x"), (NULL), (NULL);

-- 4. Refresh statistics
ANALYZE TABLE mysql_bug_mre;

-- 5. The Bug: MAX() returns NULL instead of 0.0 when using index
SELECT MAX(c0) AS wrong_result FROM mysql_bug_mre WHERE c0 = 0.0;

-- 6. Diagnostic: Force index vs Ignore index
SELECT MAX(c0) FROM mysql_bug_mre FORCE INDEX (i_c0) WHERE c0 = 0.0; -- Returns NULL (Wrong)
SELECT MAX(c0) FROM mysql_bug_mre IGNORE INDEX (i_c0) WHERE c0 = 0.0; -- Returns 0.0 (Correct)

-- 7. Diagnostic: Select shows NULL leaking into the result set
SELECT c0 FROM mysql_bug_mre WHERE c0 = 0.0; -- Shows a NULL row at the end
```
## Actual Results
- `SELECT MAX(c0) ... WHERE c0 = 0.0` returns `NULL`.

- `SELECT c0 ... WHERE c0 = 0.0` includes a row with a `NULL` value.

- The `EXPLAIN` shows a `Covering index lookup` is being used.

## Expected Results
According to SQL standard and three-valued logic, `NULL = 0.0` is `UNKNOWN`. Therefore, `NULL` values should never appear in the result set of `WHERE c0 = 0.0`.

- `MAX(c0)` should return `0.0`.

## Internal Analysis (For the developers)
The bug seems to be an off-by-one error in the index iterator. When the storage engine performs a range scan for the constant 0.0 on a FLOAT ZEROFILL column, the "stop condition" fails to trigger before reaching the NULL entries in the B-Tree. This is particularly evident when the index is "covering" the query, as the engine reads the value directly from the index leaf node without double-checking the record in the clustered index.
