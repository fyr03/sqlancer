# Pinolo-DML

*Differential State Testing* ——突出事务快照和状态差分。

**问题背景**：DBMS 的逻辑 bug 研究主要集中在 SELECT 查询，但 Update/Delete 也可能存在严重逻辑错误（错误删除、更新值不正确）。

**挑战**：

(1) DML 涉及状态变化，需要事务隔离；

(2) 嵌套子查询更复杂；

提出 PINOLO-DML，扩展近似关系到 DML，定义新型 mutators，并在多种 DBMS 上验证有效性。



**相关工作**： DQE



**方法：**

**事务/快照隔离机制**：确保每次执行在同一初始状态。

**DML Approximate Mutators**：

- Delete：谓词强弱化。
- Update：算子替换、谓词扩展。

**嵌套支持**：对子查询保持 superset/subset 关系。

**检测流程**：输入种子 DML → 生成 over/under-approx → 在隔离环境下执行 → 检查状态包含关系。



**挑战：**

**结果检查问题：**

- 对于 `UPDATE` 和 `DELETE` 操作，结果不是直接的查询结果，而是对数据库状态的修改。因此，如何检查这些操作是否存在逻辑错误，可能需要利用一些机制来验证这些操作后的数据库状态。
- 传统的 **虚拟列**（Virtual Columns）可以用于 `UPDATE` 操作的检查。例如，通过计算某一列的修改前后的差异，或使用触发器记录修改历史，从而间接检测操作是否符合预期。
- 对于 **`DELETE` 操作**，可以利用数据库的审计日志（如果可用）或设计一个“软删除”策略（例如设置标志位，而不是物理删除数据），然后通过检查删除后的数据状态来验证操作的正确性。

**数据库状态一致性：**

- 在进行 `UPDATE` 或 `DELETE` 操作后，数据库的状态会发生变化。如何确保查询语句的执行能精确反映数据库的当前状态，这要求我们在测试过程中追踪所有的修改操作。
- 通过引入 **数据库快照** 或 **事务回滚机制**，可以在执行 `UPDATE` 或 `DELETE` 后验证数据库状态的变化，并保证测试的准确性。每次执行 DML 操作后，可以通过快照与原始状态对比，检查是否存在异常。

**语法和语义的多样性：**

- **`UPDATE` 和 `DELETE`** 语句的语法结构与 `SELECT` 查询大不相同，包含 **条件表达式**、**连接操作**、**聚合函数**等复杂元素，因此，如何设计扩展到 DML 操作的“近似变换器”（Approximate Mutators）是一个重要问题。
